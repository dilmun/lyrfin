# Release Manager

The standing policy for managing lyrfin's releases — the "operating manual" that
sits above the mechanics in [`RELEASING.md`](RELEASING.md). `RELEASING.md`
explains *how the automation is wired*; this file states *the policy the release
manager (human or agent) follows*.

The pipeline is **already built** on release-please + GitHub Actions and it
works. The job is to *operate and maintain* it — not to redesign or rebuild it.
Before changing anything, read `RELEASING.md` and the existing workflows; keep
what works and only close concrete gaps.

There is exactly one decision: **Release now, or not yet.** Everything else is
automatic.

## The stack (already in place — do not replace)

- **release-please** maintains a standing "release PR" that bumps `Cargo.toml` +
  `Cargo.lock` and rewrites `CHANGELOG.md` from Conventional Commits. Merging that
  PR tags `vX.Y.Z`, publishes the GitHub Release with generated notes, and
  builds/attaches the binaries + `.sha256` checksums. It runs under a GitHub App
  token (`dilmun-release-bot`) so the release PR gets CI + the PR-title check.
- **Trunk-based, squash-only.** `main` is always releasable. Work happens on
  short `feat/… | fix/… | …` branches merged via PR. Squash merge uses the
  **PR title** as the commit subject, so the PR title *is* the single
  Conventional Commit release-please reads.
- **Branch protection** requires green CI (`lint` + `test` on
  Linux/macOS/Windows) and a valid PR title before merge, with `enforce_admins`
  on — the rules apply to everyone, owner included. Untested direct pushes to
  `main` are blocked, so every change arrives via a validated PR.
- **Two release paths:** the automated release-please path (default) and a
  manual tag-push fallback (`release.yml`) that refuses to build from a
  non-green commit.

## The one decision: "Release now?"

Answering "Release now?" = **merge the standing release PR** that release-please
keeps open. Until it is merged:

- keep landing Conventional Commits on `main`;
- release-please keeps the next version, changelog, and notes current on that PR;
- nothing is published.

Never ask the maintainer for a version number, changelog text, tag, or release
notes — the tooling computes them. The only question worth asking is
**"Release now?"** (i.e. "shall I merge the release PR?").

## Versioning (what the tooling actually does)

Determined entirely from Conventional Commit *types*, by release-please:

| Commit | Bump |
|--------|------|
| `feat` | minor (patch is *not* used pre-1.0 here) |
| `fix` | patch |
| `!` / `BREAKING CHANGE` | major — while 0.x, minor, noted in changelog |
| `perf`, `refactor`, `docs`, `chore`, `test`, `build`, `ci`, `style` | **no release** |

release-please bumps **only** on `feat`/`fix`/breaking. `perf`/`refactor` show in
the changelog but do **not** bump the version — they don't ship on their own.
"Bump patch if the refactor is externally observable" is **not automatable** for
a binary (no public API surface to diff), so it is not a rule: if a change is
user-visible, label the commit `fix`/`feat` accordingly.

## Commits & PR titles

Enforce Conventional Commits **at the PR title** — that is the squash subject and
the only thing that reaches `main`. `pr-title.yml` rejects malformed titles as a
required check; a bad title would otherwise silently drop the change from the
release. Because branch protection blocks untested direct pushes, the PR-title
check covers every change — no per-commit hooks needed.

Examples: `feat(player): add gapless playback` · `fix(library): resolve
duplicate detection` · `perf(render): reduce redraw latency` · `docs: update
README`.

## Changelog

Generated by release-please from commits; never hand-maintained on the automated
path. Grouped by Features / Fixes / etc. Only review the entry release-please
proposes in the release PR.

## Releases (on "Release now?" → merge the release PR)

Automatic: determine version → bump `Cargo.toml`/`Cargo.lock` → finalize
`CHANGELOG.md` → tag `vX.Y.Z` → GitHub Release with generated notes → build the
target binaries → attach tarballs **plus `.sha256` checksums** → open the next
unreleased cycle.

## Reliability

Deterministic, reproducible, idempotent, CI-driven, auditable. release-please
never re-releases an already-tagged version. Recovery: the manual tag-push path
(`release.yml`) is the fallback and gates on green CI for the tagged commit.

## Simplicity

Convention over configuration; smallest solution that lasts. **Do not** add:
milestone automation, per-commit hooks (redundant with the PR-title check), or a
second release tool. Optional and not yet added: build provenance/attestation
(`actions/attest-build-provenance`), signed tags.

## Do not ask the maintainer about

versions · release notes · changelog · tags · commit/PR formatting · SemVer.
Only ever ask: **"Release now?"** If not yet, keep preparing the next release
automatically.
